# CallCenter 移动端 App 可行性研究报告

## 1. 项目背景与目标
随着业务的发展，公司计划将目前基于 Web（React + NestJS）的 CallCenter 客户服务系统扩展至移动端，同时支持 iOS 和 Android 平台。
**核心目标**：实现原有工单管理、实时聊天、文件共享，以及最核心的 WebRTC（屏幕共享与语音通话）等功能在手机端的原生或近原生体验。

---

## 2. 整体难度评估：中等偏上 (★★★★☆)

对于纯文本展示或普通表单的 App 而言，转型移动端非常简单。但本系统由于重度依赖 **WebSocket 长连接** 和 **WebRTC 实时音视频**，在移动端的适配存在一定的技术壁垒，主要体现在以下两点：
1. **后台保活机制**：iOS 和 Android 杀后台极严，长连接在退到后台后极易断开，需要引入系统的离线推送（APNs/FCM）来唤醒。
2. **WebRTC 移动端兼容**：移动端没有完整的原生浏览器环境，必须借助底层组件（如 `react-native-webrtc`）来调用手机麦克风和实现视频流渲染，相比 Web 端会复杂很多。

---

## 3. 技术选型建议

考虑到目前的团队技术栈是 **React + TypeScript**，我们推荐以下几种跨平台开发方案：

### 方案 A：React Native (强烈推荐)
利用 React Native 框架，可以使用与目前前端极其相似的语法体系来编写移动端代码。
- **优点**：能够最大程度复用您现有的 TypeScript 接口定义、Socket.io 逻辑、状态管理（Zustand/Redux），学习成本极低。性能接近原生。
- **缺点**：UI 标签（如 `div` 需换成 `View`，Ant Design 需换成 RN UI 库）必须重写。WebRTC 需要接入专门的原生桥接库。

### 方案 B：Capacitor / 混合 H5 套壳 (开发成本最低)
直接把现在的网页装进一个 App 外壳里。
- **优点**：开发速度极快，现有代码几乎可以直接跑，UI 也无需大改。
- **缺点**：性能和手感不如原生，且在处理移动端麦克风权限、屏幕共享权限时经常遇到浏览器兼容性奇葩问题，容易卡壳。

### 方案 C：Flutter (性能最佳的跨平台)
使用 Google 的 Flutter 重构。
- **优点**：UI 极度顺滑，一套代码完美跑 iOS/Android，WebRTC 生态也不错。
- **缺点**：需要团队学习全新的 Dart 语言，现有前端代码几乎无法复用。

**结论**：如果您希望长远发展并保证体验，强烈建议使用 **React Native**。

---

## 4. 关键技术难点与对策

| 模块 | 移动端痛点 | 解决对策 |
| :--- | :--- | :--- |
| **实时通信** | 手机切后台或锁屏时，Socket.io 会被系统强制断开，导致收不到新消息。 | 接入苹果 APNs、谷歌 FCM（国内接极光/个推）的离线推送。收到推送点击后唤醒 App 并重连 Socket。 |
| **音视频 / 屏幕共享** | 移动端 WebRTC 缺乏标准化 API，且 iOS/Android 请求麦克风/相册的权限管理极其严格。 | 使用 `react-native-webrtc`，并针对 iOS `Info.plist` 和 Android `Manifest` 精细配置权限。屏幕共享在 iOS 上需要专门编写一个 Broadcast Upload Extension（难度较高）。 |
| **UI/交互适配** | 原 PC 端的双栏结构、大型表格在手机小屏幕上根本放不下。 | 必须重新由产品/UI进行移动端视角的交互重构，采用列表流、抽屉栏、底部 Tab 等经典的手机交互模式。 |
| **后端接口** | - | **无需改动！** 移动端完全可以复用现有的 NestJS API 和 ChatGateway。 |

## 5. 整体开发与上线流程

如果启动该项目，整个闭环流程大致如下：

1. **移动端产品设计 (1-2周)**
   - 重新设计适合手机屏幕的交互原型（如把 PC 端的表格改成移动端的卡片列表）。
2. **基础工程搭建 (1周)**
   - 初始化 React Native (推荐使用 Expo) 工程，配置路由、状态管理、Axios/Socket.io 客户端。
3. **核心业务开发 (3-4周)**
   - 登录注册、工单列表、工单详情、普通文字/图片聊天开发。
4. **WebRTC 攻坚战 (2-3周)**
   - 集成 `react-native-webrtc`，跑通双端语音通话。
   - （可选/延期）攻克移动端屏幕共享（如果不急，可以先让移动端只看 PC 的屏幕，移动端暂不支持主动分享屏幕，这能省下一半的工作量）。
5. **推送与权限系统接入 (1-2周)**
   - 接入第三方推送服务，处理系统相册、麦克风权限弹窗。
6. **打包与上架 (2-3周)**
   - **iOS**：注册苹果开发者账号（99刀/年），配置证书，提交 TestFlight 测试，经历严苛的 App Store 审核。
   - **Android**：国内各大应用商店（华为、小米、VIVO等）需要企业软著和各种备案认证，Google Play 审核相对直接。

---

## 6. 总结与建议

**难度总结**：对于有 React 经验的开发者来说，把日常的业务逻辑（工单、聊天）搬到手机上**难度很低**；但要搞定完美的后台推送和 WebRTC 音视频适配，存在**明显的陡峭学习曲线**。

**行动建议**：
1. **后端优势**：由于您的 NestJS 后端架构写得非常标准，完全做到前后端分离，所以做移动端时**后端代码几乎一行不用改**，这是一个巨大的优势。
2. **MVP 策略**：建议第一期（MVP）版本**削减 WebRTC 的屏幕共享功能**（或者只允许移动端观看，不允许移动端作为分享方），先解决“随时随地能看工单、回文字消息、接语音”的强需求，这样能把开发周期从几个月压缩到 1 个月左右。
3. **选择 React Native**：这将是复用现有 React 智力资产的最佳捷径。
