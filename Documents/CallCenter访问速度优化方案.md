# 访问速度优化方案

> 银信呼叫中心系统 · 网络架构优化路径规划

---

## 一、现状与问题分析

### 当前架构

```
用户浏览器
    ↓ HTTPS
Cloudflare 全球边缘节点（美国/欧洲）
    ↓ 加密隧道（Cloudflare Tunnel）
内网服务器 192.168.50.51（企业内网）
    ├── NestJS 后端
    └── MySQL 数据库

腾讯云（已有）
    ├── COS 对象存储（图片/附件，已迁移）
    └── CVM 43.137.62.159（TURN/STUN 服务器，WebRTC 用）
```

> **关键发现**：您已经在腾讯云上有了 COS 和 CVM（TURN服务器），说明腾讯云账号和基础设施已就绪。
> 唯一还留在内网的是**应用服务器（NestJS + MySQL）**，也是速度慢的根源。

### 为什么很慢？

| 瓶颈层 | 原因 | 影响 |
|--------|------|------|
| **上行带宽** | 企业/家庭宽带上行通常 10-50Mbps，前端打包约 5MB，首屏需完整传输 | 首屏白屏时间长 |
| **隧道延迟** | Cloudflare Tunnel 要求流量先出境到 Cloudflare 节点，再回到用户，每请求多绕 1-2 个来回 | API 响应慢 |
| **线路质量** | 国内访问 Cloudflare 国际节点，存在跨境网络抖动 | 不稳定，时有掉线 |
| **单点故障** | 依赖内网服务器常开，停电/重启即中断服务 | 可用性低 |

### 为什么买腾讯云 CDN 无用？

1. **入口被锁**：腾讯云大陆 CDN 节点强制要求 ICP 备案号，外国域名直接被拒，连配置界面都进不去。
2. **源站不兼容**：CDN 需要回源到一个固定公网 IP，Cloudflare Tunnel 是动态加密隧道，无法作为 CDN 回源地址。
3. **结论**：现有架构下购买 CDN 资源包等于浪费，问题的根本是**架构**，不是带宽资源包。

## 三、最优路径（分阶段实施）

### 🟢 阶段一：立竿见影（现在即可实施，零成本）

**目标**：最大化现有架构的性能，无需迁移服务器。

#### 1. 启用 Gzip 压缩（✅ 已完成）
前端 5MB 主包压缩后约 1.5MB，节省 70% 传输量。在 Nginx 配置中添加：
```nginx
gzip on;
gzip_types text/plain application/javascript application/json text/css;
gzip_min_length 1024;
gzip_comp_level 6;
```

#### 2. 配置静态资源强缓存（✅ 已完成）
JS/CSS 文件加上哈希名称后，可设置超长缓存，用户第二次访问完全不需要下载：
```nginx
location /assets/ {
    expires 1y;
    add_header Cache-Control "public, immutable";
}
```

#### 3. 图片/附件已托管在 COS（✅ 已完成）
您的图片和附件已经在腾讯云 COS，用户加载附件不经过您的内网，这部分已经是最优状态。

**预期效果**：二次访问速度提升约 60-80%，首次访问提升约 30%。

### 🔵 阶段二：核心优化（推荐尽快实施，约 ¥50-100/月）

**目标**：彻底解决速度慢和稳定性差的问题，无需 ICP 备案。

#### 方案：迁移后端至腾讯云香港轻量服务器

**新架构：**
```
用户浏览器
    ↓ HTTPS（直连，无隧道）
腾讯云香港轻量服务器（公网IP）
    ├── Nginx 反向代理
    ├── NestJS 后端
    └── MySQL 数据库

腾讯云（已有，保持不变）
    ├── COS 对象存储（图片/附件）← 香港服务器经内网访问，极速
    └── CVM 43.137.62.159（TURN服务器）← 同在腾讯云，WebRTC延迟更低
```

> [!TIP]
> 迁移后，NestJS 应用服务器、COS 存储、TURN 服务器三者**都在腾讯云网络内**，互相通信走腾讯内网专线，彻底消除跨公网的延迟损耗。WebRTC 音视频质量也会同步提升。

**DNS 变更（5分钟完成）：**
```
Cloudflare DNS 控制台：
  旧：CNAME → xxxx.cfargotunnel.com（Tunnel）
  新：A     → 香港服务器公网IP
```

**推荐配置：**
| 规格 | 价格 | 说明 |
|------|------|------|
| 腾讯云香港轻量 2核4G | ¥50-80/月 | 适合当前系统规模 |
| 带宽 | 建议 30Mbps | 并发 10 人同时使用绰绰有余 |

**为什么选香港？**
- 大陆访问香港延迟约 **10-30ms**，远优于 Cloudflare 国际路线的 200-500ms
- **无需 ICP 备案**，您现有的外国域名直接可用
- 腾讯云香港到 COS（广州）有内网专线，附件访问极速
- 服务器 24 小时运行，彻底摆脱内网单点依赖

**迁移工作量：**
- 修改 `sync.sh` 中的目标 IP → 10 分钟
- 在新服务器重新安装 Node.js/PM2/Nginx → 30 分钟
- 恢复数据库备份 → 5 分钟
- 修改 DNS A 记录 → 5 分钟，5-10 分钟生效

**预期效果**：首屏加载 < 2 秒，API 响应 < 100ms，可用性 > 99.9%。

### 🟡 阶段三：极致优化（可选，需 ICP 备案，约 2-3 周审核）

**目标**：在阶段二基础上，利用大陆 CDN 节点进一步提速。

#### 前提
完成 ICP 备案（在腾讯云后台提交，约 2-3 周）。备案期间阶段二的服务正常运行，不受影响。

#### 方案：前端静态文件 COS + 大陆 CDN 加速

```
首屏资源加载：
用户 → 腾讯云CDN大陆节点（缓存命中）→ 直接返回  ← 毫秒级
用户 → 腾讯云CDN大陆节点（未命中）→ COS → 返回  ← 极快

API 请求：
用户 → 香港服务器（直接连接）← 保持阶段二不变
```

**操作：**
1. 将 `frontend/dist/` 上传到 COS，开启静态网站托管
2. 绑定子域名（如 `static.yourdomain.com`）到 COS，开启 CDN
3. Vite 构建配置加入 CDN base URL，静态资源走 CDN，API 继续走香港服务器

**资源包选购（此时才有意义）：**
- 类型：**流量包**（因为前端 JS 文件较大，属于大文件少请求场景）
- 规格：境内 **100GB/年** 已完全够用（小系统每月约 1-5GB）

**预期效果**：首屏加载 < 500ms，媲美一线 SaaS 产品体验。

## 五、风险与注意事项

> [!IMPORTANT]
> 阶段二迁移前，务必先在本地或新服务器完整测试一遍系统，确认数据库和 COS 配置正常后再切换 DNS。

> [!NOTE]
> 迁移到香港服务器后，COS 存储桶建议保留在广州，香港服务器访问广州 COS 走腾讯内网专线，速度不受影响。

> [!WARNING]
> ICP 备案期间，域名对应的服务器必须是境内服务器。如果您未来想做境内备案，需要届时把服务迁回大陆服务器（但香港的数据可直接迁移）。
