# CallCenter 系统多维搜索模块架构分析报告

通过对整个系统源代码体系的结构和实现逻辑进行深入溯源，目前该系统中并存 **4 套** 在不同时期、为不同业务编写的搜索机制，它们在体验和底层技术栈上存在着极大的差异。以下是它们的深度技术解剖和对比：

---

## 1. 原生模块级搜索分析

### A. 工单广场搜索 (Tickets Search)
- **业务位置**： `/tickets` -> 工单列表顶部。
- **技术实现**： 纯 MySQL 方案。使用 TypeORM 生成复杂的 SQL：
  ```sql
  WHERE (ticket.title LIKE '%关键字%' 
      OR ticket.description LIKE '%关键字%' 
      OR ticket.ticketNo LIKE '%关键字%' 
      OR EXISTS (SELECT 1 FROM messages m WHERE m.ticketId = ticket.id AND m.content LIKE '%关键字%'))
  ```
- **架构研判**： 
  - 这是早期为了满足全貌搜索采用的最“暴力”的手段。
  - **致命隐患**：为了让搜索工单同时包含工单底下的聊天记录，使用了底层的 `EXISTS (SELECT ...)` 子关联查询。在工单积累到上万、聊天记录几十万条后，每次用户在这个输入框打字按下回车，都会引发一次巨大的 MySQL 全表扫描 (Full Table Scan) 与锁表，拖垮整个后台的所有其他正在排队存入数据的请求。

### B. 知识库搜索 (Knowledge Base Search)
- **业务位置**： `/knowledge` -> 知识库页面顶部。
- **技术实现**： 纯 MySQL 方案。
  ```sql
  WHERE doc.title LIKE '%词%' OR doc.content LIKE '%词%' OR doc.tags LIKE '%词%'
  ```
- **架构研判**：
  - **体验割裂**：因为不支持中文自然语义分词。假设客户遇到问题，输入 `脱机报错修复`，但库内的文档是 `脱机主机警告及解决`，那么 MySQL 将直接返回“0条结果”。这种“字面机械匹配”非常不适合大规模文档库的检索。
  - 此外也不支持标红摘要和高亮截取，知识库在显示超长 Markdown 时只能在外面展示标题。

### C. BBS 论坛搜索 (BBS Post Search)
- **业务位置**： `/bbs` -> 圈子/帖子广场。
- **技术实现**： 纯 MySQL 方案。只匹配标题和正文部分字符。
- **架构研判**：
  - 相比于前两者，BBS 通过“先选中板块名称”、“结合分页限制”等前置漏斗，缩小了 `LIKE`  语法的匹配基数，因此没有引发严重的性能隐患。但由于依旧不支持倒排索引，其速度和智能匹配能力停留在传统管理系统的水平。

---

## 2. 全新重构架构级搜索

### D. 增强型全站搜索 (Global Search / Elasticsearch) [新集成]
- **业务位置**： `MainLayout` 全局顶部导航栏内 / `/search` 统一落地页。
- **技术实现**： **独立高吞吐搜索引擎 (Elasticsearch 8.x + IK 中文智能分词器)**。
  - **同步机制**：底层的 `SearchSubscriber` 切片拦截了上述所有 MySQL 修改行为，并在异步时差（毫秒级）将剥离后的文本注入独立的 ES 节点。
  - **检错机制**：采用了倒排索引结合 `multi_match` 查询格式。将所有杂乱异构（Ticket/Post/Message/Knowledge）的模型融合计算相关性得分。
- **架构研判**：
  - 完全脱离了主数据库运算，真正做到了“读写分离”，即便并发再多的全局搜索也不会对 MySQL 的录入有一丁点影响。
  - 带回了全 HTML 的 `<em style="color:red">` 富文本语法高亮片段，且对拼写错误、语义模糊（“软件系统” 等价于命中 “系统软件”）极具韧性。

---

## 3. 多维度评测矩阵大盘

| 评测维度 | 基于原生 MySQL (A, B, C 模块) | 基于 Elasticsearch 全局搜索 (D 模块) |
| :--- | :--- | :--- |
| **检索延时** | ~500ms 至 3000ms+ (随时间衰退) | **< 20ms** (恒定区间内) |
| **占用系统资源** | 极高，消耗 MySQL 内核的排序计算与 I/O 读取 | 独立 JVM 运行，低开销 |
| **分词理解能力** | 无 (需要连续精确字符串匹配) | **具有生命力**，懂拼音、缩写，近义词 |
| **体验表现层** | 用户仅能看到列表，需要自己二次寻找段落位置 | 搜索结果直接截出答案上下文文段并高亮关键字 |
| **数据孤岛** | 各个模块孤立，搜工单必须去工单页 | 四个模块一网打尽，融合透传检索 |

---

## 4. 🚀 架构优化与演进建议

由于在本次实施中我们只是“新增”了一个强劲的引擎（全站搜索），因此那前面**三个“遗留”的局部模块框其实还是过去的旧逻辑引擎**。为了保持性能和体验彻底一致：

1. **废弃局部搜索中的 Like 查询（前端重定向）**  
   建议在用户在 BBS / 知识库 / 工单的顶部搜索框敲击回车时，不再发送给本模块的 `?keyword=xx` 控制器，而是由前端直接强制 **Redirect 跳转** 至 `/search` 全局搜索页面，并默认选中对应的标签（比如：打开 `/search?q=关键字&type=knowledge` 这种方式），以此将流量集中抽走，用全站搜索页面统管一切跨表搜索。
2. **优化业务后端 API 委托模式（后端代理搜索）**  
   对于某些无法带离界面操作的地方（例如：工单列表还需要和本地的 `状态`、`日期筛选器` 相互叠加强联动的场景），建议剥离工单后端的 MySQL `LIKE`，改由 `TicketsService.findAll` 内部：**“遇到 keyword -> 优先向 SearchService 或者 ES API 请求命中工单 ID 的数组集合 -> 将这个集合代入 `IN (id1, id2)` 交回给 MySQL ”** 这种微服务式的数据拉回操作。这能让局部的关联筛选既快速又有准确分词。
3. **加入细粒度权限过滤 (ACL indexing)**  
   知识文档未来会有“保密级”，当前 ES 的设计是将一切无差别索引进公域检索，下一步更新时，建议在向 Elasticsearch 输入的 JSON `body` 中挂载类似 `visibleRole: ['admin', 'tech']`，由搜索端直接拦截非匹配权限人的搜索结果，防止越界获取文档题目。
