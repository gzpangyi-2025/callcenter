# 全局模块测试覆盖补充计划

刚才我执行了后端的覆盖率测试检查 (`npm run test:cov`)，并初步了解了前端的测试配置。

## 问题概况与挑战

通过测试覆盖率报告，我们发现 `callcenter` 的后端拥有多达 16 个模块，但其中绝大多数的测试覆盖率为 **0% 或极低**（例如 `bbs`, `category`, `infrastructure`, `knowledge`, `report` 均为 0%；`ai`, `backup`, `search`, `audit`, `files`, `tickets`, `chat` 在 2%~15% 之间）。

> [!CAUTION]
> **工作量预警**：为现有近 20 个模块（包含控制器、服务类、网关等数十个复杂类，涉及数据库和外部 API 模拟）一次性补齐全量单元测试，将涉及数千行代码的编写。如果一次性全部执行，会导致耗时极长且难以保证测试质量。

为了保证测试的有效性和系统稳定性，我强烈建议采用**分阶段（Phased Approach）**的方式来逐步推进测试覆盖。

## 提议的分阶段实施方案

我建议按照系统的核心重要程度，分批次为模块补充测试：

### 阶段 1：核心权限与用户模块 (Phase 1: Core Identity)
这些是系统的基石，保证安全不被破坏。
*   `src/modules/users` (用户管理)
*   `src/modules/roles` (角色与权限)
*   `src/modules/auth` (登录认证与 JWT)

### 阶段 2：核心业务模块 (Phase 2: Core Business)
系统最核心的功能逻辑。
*   `src/modules/tickets` (工单系统)
*   `src/modules/knowledge` (知识库)
*   `src/modules/chat` (聊天与 WebRTC)

### 阶段 3：基础设施与配置 (Phase 3: Infrastructure)
*   `src/modules/files` (文件与 COS 存储)
*   `src/modules/settings` (系统设置)
*   `src/modules/backup` (数据备份)

### 阶段 4：边缘功能与高级模块 (Phase 4: Advanced Features)
*   `src/modules/ai` (Codex 及其他 AI 功能)
*   `src/modules/bbs`, `category`, `report`, `search`, `audit`

### 阶段 5：前端测试 (Phase 5: Frontend)
*   为 React 组件、Store 和 API 请求编写 Vitest 测试。
