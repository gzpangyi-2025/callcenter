# 屏幕共享一键截图与暂存功能可行性研究报告

## 1. 需求解析
- **核心目标**：在屏幕共享（观看/远程协助）过程中，提供快捷截图功能，方便工程师留存关键故障画面。
- **功能流转**：
  1. 点击一键截图。
  2. **最新一张截图自动覆盖并保存到操作系统剪贴板**，方便工程师随时、即刻粘贴到外部软件（如发现前面截图错误，再次截图即可自然覆盖掉剪贴板的旧错图）。
  3. 截图同时会被加入到前端内存临时容器中（最大容量 10 张，页面刷新则清空），**不提前消耗服务器存储资源**。
  4. 容器内图片支持：预览、**删除**、下载（下载后清理内存）、简单的图像编辑（标注等）。
  5. **正式落盘**：仅当用户在容器中点击“发送至聊天”时，才将截图上传到服务端 OSS 并作为聊天记录永久留存。

## 2. 技术可行性评估

### 2.1 视频流截图提取 (高可行性)
**技术方案**：
利用 HTML5 的 `<canvas>` 绘图 API。当用户点击截图时，获取当前的 `<video>` DOM 元素。在内存中创建一个离屏的 `<canvas>`，尺寸设置为视频流的实际分辨率。通过 `context.drawImage(videoElement, 0, 0, width, height)` 方法提取视频的当前帧，随后通过 `canvas.toBlob()` 生成图片二进制数据。
**风险点**：此为 Web 侧处理音视频流的成熟标准方案，无明显风险。

### 2.2 自动写入系统剪贴板 (中高可行性)
**技术方案**：
使用现代浏览器的 `navigator.clipboard.write()` API，将截图的 Blob 封装为 `ClipboardItem` 写入。
**风险点与兼容性限制**：
- **安全上下文**：该 API 强制要求在安全上下文（HTTPS 或 `localhost`）下运行。如果系统部署为 HTTP 且非 `localhost`，此功能会被浏览器屏蔽（不过目前本系统应该符合要求）。
- **用户交互验证**：API 必须在由用户主动触发的操作（如点击按钮）的同步代码流中执行。这非常符合我们“点击截图按钮”的场景。

### 2.3 临时容器状态管理 (高可行性)
**技术方案**：
利用 React 的内部状态（如 `useState` 或 `zustand`）维护一个包含文件数据及时间戳的数组：`Array<{ id: string, blob: Blob, objectUrl: string, timestamp: number }>`。
- **内存消耗**：保存 10 张 1080P PNG 截图，占用内存大约在 30MB 内，对于现代 PC 浏览器而言完全在安全和可控的性能范围内。
- **自动管理**：生成 `Blob` 后使用 `URL.createObjectURL` 提供给界面预览，并在图片被移出容器时调用 `URL.revokeObjectURL` 释放内存，防止内存泄漏。

### 2.4 容器内操作与分发 (高可行性)
- **下载清理**：利用临时的 `<a>` 标签加上 `download` 属性，把 Blob 对应的 URL 赋给它触发下载，回调中执行状态剔除。
- **图片编辑（标注）**：可通过引入前端轻量级画板组件（如 `react-konva`）实现。用户在画布上添加红框、箭头后，再次通过 `toBlob` 导出新的图片数据。这是开发工作量相对集中的环节。
- **发送至聊天（持久化）**：将暂存的 Blob 包装成 `File` 对象，复用当前系统现成的 OSS 文件上传接口 `filesAPI.upload`，上传获取 URL 后，将包含图片 URL 的消息载体通过 Socket 发送即可。

## 3. 交互与 UI 设计建议

1. **截取按钮**：在屏幕共享画面的底部控制栏中新增一个明显的 `[✂️ 截图]` 按钮。
2. **交互反馈**：点击截图时，播放“咔嚓”音效或在视频区域加一层 100ms 的白色半透明遮罩闪光动画，同时气泡提示“截图已复制到剪贴板”。
3. **临时容器展现**：在控制栏右侧（或者界面侧边悬浮）增加一个“截图暂存箱”图标，图标右上角带徽标数字（如 `3/10`）。点击后向上方展开一个 Popover 气泡面板，使用横向或网格布局展示缩略图。
4. **溢出处理机制**：当数量达到上限 10 张且用户继续截图时，建议采用**先进先出（FIFO）**策略，自动顶掉最早的一张截图，保证用户操作的流畅性，而不需要打断用户去手动清理。

| **阶段二** | UI 层面：截图按钮、快门动画反馈、悬浮暂存箱面板与缩略图列表渲染 | 0.5 天 |
| **阶段三** | 交互逻辑：单张图片的预览、直接下载（并销毁状态）、一键推送到工单聊天室并上传至 OSS | 0.5 天 |
| **阶段四** | **可选进阶**：在发送前增加前端画布，实现画笔、红框、箭头等简单的图片批注功能 | 1.0 天 |

## 5. 结论

**可行性极高。**
该需求在技术上是 100% 可实现的，并且是典型的“纯前端驱动”功能，不需要后端进行任何接口层面的配合开发（发送功能完全可以复用现有的图片上传和聊天逻辑）。
它能极大提升技术支持人员在远程排障时的“取证”和记录效率。如果认可此方案，可以随时开始第一阶段的代码实施。
