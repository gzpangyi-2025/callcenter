## 二、详细发现

### 🔒 安全类 (Security)

#### SEC-01: JWT Secret 硬编码兜底值 [中风险]
- **文件**: [jwt.strategy.ts](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/auth/jwt.strategy.ts#L22)
- **问题**: `secretOrKey: configService.get('JWT_SECRET') || 'fallback_secret'`。如果环境变量未配置，系统将使用可预测的固定密钥，攻击者可伪造任意 JWT。
- **建议**: 启动时校验 `JWT_SECRET` 是否存在，若缺失则拒绝启动并抛出致命错误。

#### SEC-02: WebSocket CORS 硬编码 localhost [中风险]
- **文件**: [chat.gateway.ts:L24](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/chat/chat.gateway.ts#L24)
- **问题**: `origin: ['http://localhost:5173', 'http://localhost:3001']` 硬编码在代码中。生产环境下仅因 Nginx 反代掩盖了此问题，但如果攻击者直连后端端口，WebSocket 握手将失败。与 `main.ts` 中使用 `CORS_ORIGIN` 环境变量的方式不一致。
- **建议**: 统一从环境变量读取 CORS 来源。

#### SEC-03: 文件静态服务缺少 JWT 认证 [中风险]
- **文件**: [files.controller.ts:L64](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/files/files.controller.ts#L64)
- **问题**: `GET /files/static/:filename` 不需要任何认证即可访问。虽然文件名是 UUID 不可枚举，但如果攻击者获得了文件名（如通过聊天记录 XSS），可以直接下载敏感附件。
- **建议**: 对敏感文件类型添加认证，或设置 COS 预签名链接的超短有效期。

#### SEC-04: 搜索接口缺少 PermissionsGuard [低风险]
- **文件**: [search.controller.ts](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/search/search.controller.ts#L5-L6)
- **问题**: 仅有 `AuthGuard('jwt')`，没有 `PermissionsGuard`。任何已登录用户（包括权限最低的普通用户）都可以执行全文搜索和 `POST /search/sync` 触发全量索引重建。
- **建议**: 补充 `PermissionsGuard`，搜索接口添加 `@Permissions('tickets:read')`，sync 接口添加 `@Permissions('admin:access')`。

#### SEC-05: `PermissionsGuard` 中 username 绕过风险 [低风险]
- **文件**: [permissions.guard.ts:L38](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/auth/permissions.guard.ts#L38)
- **问题**: `user.role.name === 'admin' || user.username === 'admin'`。按 username 判断超管特权，如果有人注册了名为 `admin` 的非管理员账户（虽然当前系统有唯一约束），理论上可以绕过权限。
- **建议**: 仅通过 `user.role.name` 判断超管权限，移除 `user.username` 的判断逻辑。

### 🏗️ 架构类 (Architecture)

#### ARCH-01: ChatGateway 单文件过大 (860 行)
- **文件**: [chat.gateway.ts](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/chat/chat.gateway.ts)
- **问题**: 聊天消息、屏幕共享信令、语音通话信令、房间锁定逻辑、在线用户管理全部堆在一个文件中，职责过多。
- **建议**: 拆分为 `ChatGateway`（消息+房间）、`ScreenShareGateway`（屏幕共享信令）、`VoiceGateway`（语音信令），或至少提取 Service 层。

#### ARCH-02: BBS Controller 内联权限校验
- **文件**: [bbs.controller.ts:L125-L157](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/bbs/bbs.controller.ts#L125-L157)
- **问题**: `update`、`remove`、`batchRemove` 方法中手动在 Controller 层构建 `canBypass` 逻辑进行权限判断，而不是利用 Guard 或 Service 层统一处理。这导致权限逻辑分散、重复、且容易在新增接口时遗漏。
- **建议**: 将权限判断下沉到 Service 层或专门的 Guard，保持 Controller 精简。

#### ARCH-03: 内存级状态管理无持久化
- **文件**: chat.gateway.ts 中的 `onlineUsers`、`activeSharers`、`activeVoiceRooms`
- **问题**: 所有在线状态、屏幕共享状态、语音通话状态仅保存在进程内存中。如果使用 PM2 cluster 模式或有多实例部署，状态将不同步。
- **建议**: 当前单实例部署可接受，但如果未来扩展需考虑引入 Redis Adapter。

### 🔧 类型安全 (Type Safety)

#### TYPE-01: 大量 `as any` 类型断言 [高]
- **代表文件**: chat.gateway.ts (47 处)、bbs.controller.ts、tickets.controller.ts
- **问题统计**:
  - `@typescript-eslint/no-unsafe-member-access`: **325 处**
  - `@typescript-eslint/no-unsafe-assignment`: **192 处**
  - `@typescript-eslint/no-unsafe-argument`: **116 处**
  - `@typescript-eslint/no-unsafe-call`: **54 处**
  - `@typescript-eslint/no-unused-vars`: **32 处**
- **核心根因**: WebSocket `client` 对象通过 `(client as any).userId` 访问自定义属性，绕过了 TypeScript 类型检查。项目已定义了 `AuthenticatedSocket` 接口，但 Gateway 中并未一致使用。
- **建议**: 在 Gateway 方法入口统一 `const authClient = client as AuthenticatedSocket`，后续全部使用类型安全的属性访问。

#### TYPE-02: Controller 参数 `@Body() body: any`
- **文件**: bbs.controller.ts 多处、infra.controller.ts
- **问题**: 请求体类型为 `any`，绕过了全局 `ValidationPipe` 的验证。
- **建议**: 为每个接口定义严格的 DTO 类并配合 `class-validator` 装饰器。

### ⚡ 性能类 (Performance)

#### PERF-01: 文件迁移使用同步 I/O 阻塞事件循环
- **文件**: [files.controller.ts:L170](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/files/files.controller.ts#L170)
- **问题**: `fs.readFileSync(filePath)` 在 `setImmediate` 回调中执行，虽然不阻塞请求响应，但在实际执行期间会阻塞 Node.js 事件循环，影响其他请求的处理。
- **建议**: 改用 `fs.promises.readFile` 或流式上传。

#### PERF-02: 每次发消息查一次完整 Ticket
- **文件**: [chat.gateway.ts:L555-L576](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/chat/chat.gateway.ts#L555-L576)
- **问题**: 每次 `sendMessage` 都要 `ticketRepository.findOne` 查出完整 Ticket（含 participants 关联）来确定通知目标。在高频聊天场景下，这会频繁命中数据库。
- **建议**: 对工单参与者列表做进程内缓存（内存 Map + TTL），或在 joinRoom 时预加载。

#### PERF-03: WebSocket CORS 写死在代码中
- 与 SEC-02 重叠。此外，`origin` 数组过短，如果未来增加多个前端域名需要改代码重新编译。

### 🧪 测试覆盖 (Testing)

#### TEST-01: 测试覆盖率极低
- **现状**: 整个后端仅有 **1 个测试文件** (`tickets.service.spec.ts`)，包含约 6 个测试用例，覆盖了 `findOne`、`deleteTicket`、`batchDelete`、`assign` 的基本路径。
- **缺失**: 
  - 无 E2E 测试
  - 无认证/授权相关测试（PermissionsGuard、JWT 验证流程）
  - 无 WebSocket 集成测试
  - 无文件上传测试
  - 无前端单元测试
- **建议**: 优先为关键安全路径（认证、权限守卫）和核心业务逻辑（工单状态机、消息发送）编写测试。

### 📝 错误处理 (Error Handling)

#### ERR-01: 静默 catch 块
- **文件**: tickets.service.ts (`/* ignore */` at L589, L651)
- **问题**: 文件删除失败时的异常被完全忽略，不记录日志。生产环境可能导致残留文件永远无法被发现和清理。
- **建议**: 至少记录一条 `this.logger.warn()` 日志。

#### ERR-02: 使用 `console.log/warn/error` 而非 NestJS Logger
- **文件**: files.controller.ts (L184, L187)、main.ts (L54)
- **问题**: 项目其他地方已统一使用 `Logger`，但这三处使用了原生 `console`，导致日志输出格式不一致且无法被 NestJS 日志管道（如 Winston/Pino）捕获。

#### ERR-03: `@typescript-eslint/no-floating-promises` (16 处)
- **问题**: 有 16 处异步 Promise 调用没有 `await` 也没有 `.catch()`，如果这些 Promise 被拒绝，将产生 `UnhandledPromiseRejection`，可能导致 Node.js 进程崩溃。

### 📦 其他代码质量问题

#### MISC-01: Category Controller 没有统一的 Guard 注册
- **文件**: [category.controller.ts](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/category/category.controller.ts#L16-L17)
- **问题**: 没有在类级别注册 Guard，而是在每个方法上单独挂载 `@UseGuards`，风格与其他 Controller 不一致。
- **建议**: 统一为类级别 `@UseGuards(AuthGuard('jwt'), PermissionsGuard)`。

#### MISC-02: 未使用的导入
- **检测**: `@typescript-eslint/no-unused-vars` 有 **32 处**。
- **建议**: 批量清理未使用的 import 和变量。

#### MISC-03: 暴露的调试接口
- **文件**: [tickets.controller.ts:L80-L83](file:///Users/yipang/Documents/Antigravity/callcenter/backend/src/modules/tickets/tickets.controller.ts#L80-L83)
- **问题**: `GET /tickets/test-reload` 是一个调试接口，没有任何权限控制。
- **建议**: 移除或至少加上 `@Permissions('admin:access')` 保护。

## 五、总结

项目整体架构设计合理，模块划分清晰，业务功能完整。经过前期的 P0/P1/P2 安全修复后，最核心的越权漏洞已被堵住。当前主要短板集中在 **类型安全** 和 **测试覆盖** 两个方面：

1. 大量的 `as any` 使得 TypeScript 的类型检查形同虚设，容易引入运行时错误
2. 测试几乎空白，任何重构或新增功能都缺乏安全网
3. 安全层面仍有几处中低风险项需要收口

建议按照上述四个 Phase 的优先级顺序，逐步推进整改工作。
